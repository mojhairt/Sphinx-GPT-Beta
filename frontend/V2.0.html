Analysis
Current Issues & Weaknesses
What makes v1 scientifically shallow — and the exact fix for each problem
Critical
Issue #1 — Classifier uses a full LLM call
Location: Role 0 — classify_problem() in llm_manager.py

Every single user message fires a Groq API call purely to decide "math or chat?" — that's wasted tokens, added latency (~800ms), and API cost on the most trivial decision in the system.

Root cause: LLM used where a lightweight local model should handle it.
Fix: Fine-tuned bert-base-uncased running locally — 10× faster, zero API cost, higher accuracy.
Critical
Issue #2 — No persistent memory
Location: Role 6 — chat(), history parameter

Conversation history is passed manually per request and evaporates on every page refresh. Users repeat themselves every session. There's zero continuity, zero personalization, zero learning about the user over time.

Fix: Short-term session memory (in-process) + long-term ChromaDB vector store tracking weak topics, mistakes, and preferences per user.
Medium
Issue #3 — chat_with_math() context is fragile
Location: chat_with_math() in llm_manager.py

Math results are injected as raw text inside the user message. The LLM sometimes ignores the provided answer and re-solves from scratch, producing inconsistent responses.

Fix: Inject math result via system prompt or structured assistant turn — not raw user text.
Medium
Issue #4 — Fixed max_tokens=1024 for all roles
Location: _call_llm() and _call_chat()

The Classifier needs ~50 tokens. The Step Generator needs 800+. Both get the same 1024 — wasteful for one, potentially truncating for the other.

Fix: Per-role dynamic token limits configured in LLMManager.
Medium
Issue #5 — No smart fallback on API errors
Location: _extract_json(), _call_llm() error handling

If Groq fails or JSON parsing breaks, a raw RuntimeError surfaces to the user. Zero graceful degradation, zero retry logic, zero user-friendly messaging.

Fix: Retry logic, fallback LLM switch, and friendly error messages at every failure point.

Core Engine Development
Four deep technical pillars that transform the system's intelligence from the ground up
Pillar 1 — NLP
BERT Classifier — Replace Role 0
Fine-tune bert-base-uncased on a labeled math/non-math sentence dataset. The model outputs a structured prediction: { is_math, branch, problem_type, confidence }. It's a drop-in replacement for the current LLM classifier — no other code changes needed beyond swapping the function call.

Training pipeline: Collect dataset → tokenize with BertTokenizer → freeze base layers → train classification head → evaluate on held-out test set → export to ONNX for fast inference.

Result: 80ms local inference vs 800ms API call. Zero API cost. Higher accuracy. Confidence scores for uncertainty handling.

Pillar 2 — Agent Architecture
AI Agent — ReAct Pattern (from scratch, no LangChain)
Replace the hardcoded pipeline in app.py with a ReAct (Reasoning + Acting) loop. The agent decides which tool to invoke, observes the result, updates its reasoning, and repeats — until it constructs the final answer autonomously. Built from scratch for full academic transparency and zero framework overhead.

Each existing Role becomes a Tool: steps_tool, hints_tool, ocr_tool, word_problem_tool, classify_tool, study_tool.

Pillar 3 — Memory
Memory Layer — Short-term + Long-term
Short-term: Full session conversation history kept in-memory per request — injected into every LLM call automatically.

Long-term: ChromaDB vector store — each interaction is embedded with sentence-transformers and stored. Before every response, semantically similar past interactions are retrieved and injected as context. Tracks: weak topics, repeated mistakes, explanation style preferences, topic history.

Result: The system knows you. It remembers you struggled with integration last week. It adjusts difficulty. It references your past work.

Pillar 4 — Reliability
Multi-LLM Router + Fallback Chain
Instead of one fixed Groq key for everything, route each task to the best available model. The router selects based on task type, token budget, and API availability.

Routing logic: Step generation → Groq Llama 3.3-70B (speed). Complex word problems → GPT-4o (reasoning). Vision/OCR tasks → Gemini. If primary fails → automatic fallback to next available. If all fail → graceful error with retry scheduling.

Solves Issue #5 completely and makes the system resilient to API outages.

Signature Feature
Study Mode — Socratic AI Tutor
Not a Q&A bot. A real tutor that follows your thinking, challenges you, adapts to you, and tracks your progress over time.
What makes this different
Classical chatbots answer questions. Study Mode asks them. It follows the Socratic Method — guiding students to discover answers through targeted questions, progressive hints, and reflection. This is a well-established pedagogical framework backed by decades of educational psychology research. Your professor will recognize it.
Session Flow
01
Topic Input & Memory Retrieval
User enters: "I want to study Integration by Parts." Agent immediately queries the Memory Layer — retrieves past sessions on this topic, previous mistakes, weak spots, and the user's preferred explanation style. Session is already personalized before it begins.
02
Level Assessment (2–3 diagnostic questions)
Agent calibrates difficulty through targeted diagnostic questions. Adjusts the entire session based on answers — no time wasted on what the user already knows. Identifies the exact knowledge gap to target.
03
Socratic Questioning Loop
Agent never gives the answer directly. It asks: "What do you think the first step is?" → Wrong answer → Hint (vague) → Try again → Hint (medium) → Try again → Near-answer hint → Correct → Harder question. Exactly like a real tutor who refuses to just hand you the answer.
04
Pomodoro Timer Integration
25-minute focused study blocks with 5-minute breaks. At each break, the agent auto-generates a personalized mini-summary: "Here's what you mastered today. Here's what we tackle next. Here's a pattern I noticed in your mistakes."
05
Session Report + Memory Update
End of session: full progress report — topics covered, correct/incorrect ratio, identified weak points, time spent, recommended next session. All stored to the Memory Layer. Next session starts informed.
